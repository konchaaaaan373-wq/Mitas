# MITAS DB 要件定義

## 1. 目的

MITAS のデータベースは、医療機関からの相談を受けて終わるためのものではなく、相談から要件整理、人材充足、進捗管理、振り返りまでを継続的に支える業務基盤とする。

## 2. 業務で管理したい対象

### 2.1 問い合わせ

- どの施設種別から
- どの地域で
- どの程度急いで
- 何の支援を求めているか

### 2.2 顧客情報

- 法人
- 施設
- 担当者
- 過去案件

### 2.3 欠員・充足案件

- どの施設で
- どの職種を
- 何人
- どの期限までに
- どの要件で充足したいか

### 2.4 候補者情報

- 職種
- 専門性
- 経験年数
- 希望条件
- 可動時期
- 対応地域

### 2.5 マッチング進捗

- 候補提示
- 連絡中
- 面談
- 提案
- 成約
- 見送り理由

### 2.6 オペレーション履歴

- 誰がいつ対応したか
- 次に何をすべきか
- どの案件が停滞しているか

## 3. MVP で必要な機能

### 3.1 必須

- 問い合わせ保存
- 法人・施設・担当者管理
- 案件作成
- 案件ごとの要件管理
- 候補者登録
- 案件と候補者のマッチ管理
- タスク管理
- 活動履歴保存

### 3.2 あると強い

- 案件ダッシュボードビュー
- 候補者検索用インデックス
- 案件ごとのボトルネック把握
- 失注理由・見送り理由の集計

## 4. 保存しないもの

- 患者個人情報
- 診療録や介護記録
- マイナンバー等の高機密個人情報
- 法的判断そのもの

MITAS で扱うのは、採用・配置・要件充足に必要な運用情報に限定する。

## 5. 主なユースケース

### UC-01 問い合わせから案件化

1. フォームから問い合わせが入る
2. 運営が確認し、法人と担当者を登録
3. 問い合わせを案件に変換

### UC-02 欠員案件の立ち上げ

1. 施設と担当者を紐づける
2. 職種、必要人数、期限、必須条件を登録
3. 優先度とリスクを設定する

### UC-03 候補者とのマッチング

1. 候補者を登録する
2. 案件要件と候補者を紐づける
3. 選考・提案・成約まで進捗管理する

### UC-04 オペレーション管理

1. 対応ログを残す
2. 次回アクションをタスク化する
3. 停滞案件を抽出する

## 6. 主要 KPI

- 月次問い合わせ数
- 問い合わせから案件化への転換率
- 案件ごとの充足所要日数
- 要件別の成約率
- 候補者提案から面談化への転換率
- 失注理由内訳

## 7. 今回の設計判断

- 初期段階から「問い合わせ」と「案件」を分離する
- 法人、施設、担当者を分けて将来の複数拠点運用に備える
- 候補者は案件とは独立したプールとして保持する
- 進捗は `matches` と `tasks` と `activities` に分け、集計しやすくする
- PostgreSQL の enum / array / jsonb / GIN index を活用する
